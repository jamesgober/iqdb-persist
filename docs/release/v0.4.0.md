# iqdb-persist v0.4.0 — Compression + feature freeze

**The feature set is complete.** v0.4.0 adds optional Zstd / LZ4 snapshot
compression on top of the v0.2 atomic-snapshot foundation and the v0.3
write-ahead log, and **declares the persistence feature set frozen**: from
here, the road to 1.0 is `storage-io` integration plus the public-API and
on-disk-format freeze (v0.5), then hardening — no new features.

## What is iqdb-persist?

The embedded persistence layer of the iQDB vector database. It adds durable
snapshot save/load to any index that implements `iqdb_index::Index`, behind
a versioned header, a CRC32 integrity check, and an atomic write; an
optional write-ahead log makes mutations between snapshots crash-recoverable;
and, as of v0.4, snapshots can be transparently compressed.

## Compression

```rust
use iqdb_persist::{Compression, PersistConfig, PersistedIndex};

let mut cfg = PersistConfig::new("/var/lib/app/index.iqdb");
cfg.compression = Compression::Zstd { level: 3 }; // or Compression::Lz4

PersistedIndex::open_with(my_index, cfg.clone())?.save()?;       // payload compressed
let restored: PersistedIndex<MyIndex> = PersistedIndex::load(cfg)?; // transparently decompressed
```

Enable the codecs via cargo features (off by default):

```toml
iqdb-persist = { version = "0.4", features = ["zstd", "lz4"] }
```

- **`Compression::Zstd { level }`** — Zstandard via the reference C library
  (`zstd` feature); best ratio, `level` in `1..=22`.
- **`Compression::Lz4`** — LZ4 block format via the pure-Rust `lz4_flex`
  (`lz4` feature); fastest, no C toolchain.
- **`Compression::None`** — always available; payload stored verbatim.

Compression applies only to snapshots — the WAL stays uncompressed
(per-record compression of tiny mutations is not worthwhile). Selecting a
scheme whose feature is not compiled in returns
`PersistError::Unsupported` at construction, never a panic or a silent
fallback.

## On-disk format v2

The snapshot format moves to version `2`: the payload region gains a 9-byte
preamble `[scheme tag u8][uncompressed_len u64 LE]`, and the existing CRC32
covers the whole region — so corruption is caught **before** decompression.
The chosen scheme is recorded on disk, so a file decodes correctly as long
as that scheme's feature is enabled at load time.

**Backward compatible:** version-1 snapshots written by v0.2 / v0.3 still
load (as uncompressed). The reader accepts `1..=2`; the writer emits `2`.

## Highlights

- **CRC-before-decompress.** The integrity check runs on the stored
  (compressed) bytes, so a bit flip surfaces as `ChecksumMismatch` rather
  than a confusing codec error.
- **Decompression-bomb guard.** A payload claiming to expand past a
  per-file ratio bound is rejected as `InvalidPayload`. The bound scales
  with the compressed size, so it never limits a legitimately large
  snapshot.
- **No C unless you ask.** Default builds pull neither codec; `lz4` is
  pure-Rust; only `zstd` brings the C library, and only when enabled.

## Performance

`criterion` benches (`benches/compression_bench.rs`, 10 000 × 128-D payload):

- **LZ4** — compress ~15 GiB/s, decompress multi-GiB/s.
- **Zstd (level 3)** — compress ~4.8 GiB/s, decompress ~2.2 GiB/s.

(Throughput is measured over a compressible vector payload; absolute numbers
are hardware-dependent. LZ4 favours speed, Zstd favours ratio.)

## Verification

- `fmt`, `clippy -D warnings` (default and `--all-features`),
  `test` (default and `--all-features`), and `cargo doc -D warnings` are
  clean; builds on stable and the 1.87 MSRV; `cargo deny` + `cargo audit`
  clean. Verified on **Windows and WSL2/Linux** (the latter exercises both
  the `cfg(unix)` directory-`fsync` and the zstd C build).
- **Zero `unsafe`** in this crate (`#![forbid(unsafe_code)]`); no `unwrap` /
  `expect` / `todo!` / `dbg!` in shipping code.
- Tests: per-scheme snapshot round-trips, a compression-shrinks-the-file
  check, CRC-catches-corruption, a feature-disabled rejection, and a
  legacy-v1 snapshot load — plus the v0.2/v0.3 suites. All green.

## Breaking changes

**None** at the Rust API level — the additions are new `Compression`
behaviour and two cargo features. The on-disk **format moves to v2**, but v1
files still load, so existing snapshots keep working. Neither the API nor
the format is frozen yet — that lands at v0.5.

## What's next

- **v0.5.0 — `storage-io` integration, adversarial / partial-write tests,
  and the public-API + on-disk-format freeze.** The last feature-bearing
  phase before the 0.6–0.9 hardening series and 1.0.

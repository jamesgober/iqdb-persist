# iqdb-persist v0.3.0 — Write-ahead log

**Durability between snapshots.** v0.3.0 adds the write-ahead log, replay,
and crash recovery on top of the v0.2 atomic-snapshot foundation. With the
WAL enabled, every `insert` / `delete` is logged and `fsync`ed *before* it
is applied in memory, and replayed onto the snapshot on `load` — so a
mutation acknowledged to the caller survives a crash.

## What is iqdb-persist?

The embedded persistence layer of the iQDB vector database. It adds durable
snapshot save/load to any index that implements `iqdb_index::Index`, behind
a versioned header, a CRC32 integrity check, and an atomic write. v0.3 adds
the optional write-ahead log so mutations made between snapshots are
crash-recoverable, not just the snapshots themselves.

## The WAL surface

```rust
use iqdb_persist::{PersistConfig, PersistedIndex};

let mut cfg = PersistConfig::new("/var/lib/app/index.iqdb");
cfg.wal_enabled = true;

// open_with writes a base snapshot and opens a fresh WAL beside it.
let mut db = PersistedIndex::open_with(my_index, cfg.clone())?;

// Each mutation is logged + fsynced, then applied in memory.
db.insert(id, vector, metadata)?;
db.delete(&other_id)?;

// Fold the log back into a fresh snapshot (bounds WAL growth).
db.checkpoint()?;

// After a crash: load replays the WAL onto the snapshot.
let recovered: PersistedIndex<MyIndex> = PersistedIndex::load(cfg)?;
```

New in the surface:

- **`PersistConfig::wal_enabled`** — turn on the write-ahead log.
- **`PersistedIndex::insert` / `delete`** — the durable, log-before-apply
  mutation path (with rollback if the in-memory apply is rejected).
- **`PersistedIndex::checkpoint`** — snapshot the state and truncate the
  WAL.
- **`FsyncPolicy::Periodic`** now governs WAL append `fsync` cadence.

## Highlights

- **Log-before-apply.** A mutation is durable in the WAL before it touches
  memory; if the apply is then rejected (duplicate id, absent delete), the
  frame is rolled back so the log never drifts from the index.
- **Torn-tail safe.** Every frame carries its own length and CRC32. A crash
  mid-append leaves a truncated or mis-checksummed final frame, which replay
  detects and discards — that mutation was never acknowledged.
- **Portable, compact format.** Records encode `VectorId` (`U64` / `Bytes`),
  the vector, and `Metadata` in fixed-width little-endian; the same file
  reads on 32- and 64-bit hosts. A 256 MiB per-record cap bounds any
  allocation a corrupt length could request.
- **Bounded growth.** `checkpoint` compacts the log into a snapshot, so the
  WAL never grows without limit.

## Performance

`criterion` benches (`benches/wal_bench.rs`, `FsyncPolicy::Never` to isolate
library cost from disk `fsync` latency):

- **WAL append** (128-D insert): ~3.9 µs/op (~254 K inserts/s).
- **Replay** of 10 000 128-D records on load: ~3.6 ms (~2.8 M records/s).

`fsync`-per-op latency under `FsyncPolicy::Always` is disk-hardware-bound and
sits on top of these figures; `Periodic` amortises it.

## Verification

- `fmt`, `clippy -D warnings` (default and `--all-features`),
  `test --all-features`, and `cargo doc -D warnings` are clean; builds on
  stable and the 1.87 MSRV; `cargo deny` + `cargo audit` clean.
- **Zero `unsafe`** (`#![forbid(unsafe_code)]`); no `unwrap` / `expect` /
  `todo!` / `dbg!` in shipping code; bounds-checked record decoding never
  panics on a malformed log.
- Tests: WAL lifecycle + crash-recovery integration (replay, checkpoint
  no-double-apply, torn-tail discard, `Bytes` ids, metadata survival,
  continued appends after recovery), a `proptest` record round-trip, and
  the v0.2 snapshot/format suite — all green.

`loom` is **not applicable**: `PersistedIndex` is single-writer
(`&mut self` for mutation, `&self` for reads) with no shared-state or
lock-free path, so there is nothing for a concurrency model checker to
explore.

## Breaking changes

**None** at the API level relative to v0.2.0 — the additions are new methods
and a now-honored config flag. The on-disk snapshot format is unchanged
(version `1`); the WAL is a new, separate file. Neither format is frozen yet
— the freeze lands at the v0.5.0 API freeze.

## What's next

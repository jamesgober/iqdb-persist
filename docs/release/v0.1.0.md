# iqdb-persist v0.1.0 — Scaffold

**The repository bootstrap.** v0.1.0 establishes the structure, tooling, and quality gates the persistence layer is built on — no domain logic yet. It exists so the implementation lands one verified step at a time onto a foundation that already enforces the iQDB engineering standard.

## What is iqdb-persist?

The disk-persistence layer of the iQDB vector database — what moves iQDB from demo-only to actually usable. It serializes indexes and vectors to a versioned on-disk format, logs every mutation to a write-ahead log (WAL) before it touches memory, and recovers cleanly after a crash. Saves are atomic (write-to-temp-then-rename) so a partial write can never corrupt existing data, and each stored artifact carries a CRC32 integrity check. It is the embedded persistence layer and is designed to sit on the `storage-io` substrate (the renamed `fsys-rs`) rather than touching files directly. It sits above `iqdb-types` (core types) and `iqdb-index` (wraps any index as persistable).

## What's next

- **v0.2.0 — on-disk format + atomic save/load + CRC32.** The hard part, not deferred: a versioned file header with magic and per-artifact CRC32 integrity, and the atomic write-to-temp-then-rename save/load path so a partial write never corrupts existing data — with the core invariants property-tested.
- **v0.3.0 — WAL append + replay + crash recovery.** The write-ahead log: every mutation logged before it is applied in memory, replayed on startup to recover from a crash.
